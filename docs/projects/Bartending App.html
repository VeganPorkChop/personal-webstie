Overview

BartendingApp is an Android (Java) application that helps users browse classic cocktails, learn recipes, favorite drinks, and explore curated bundles. Admins can publish custom drink bundles, and users can rate and review them. The app includes a placeholder for tutorial videos and supports translating drink details into Spanish, French, or Italian.

Documentation included

This page covers the entire repo documentation including the Timeline Progress feature spec, build instructions, asset workflows (videos), monetization setup (AdMob), and admin bundle controls.

Features

  • Browse a list of standard drinks.
  • Search by drink name.
  • View details for each drink including ingredients, instructions, and a placeholder for a tutorial video.
  • Mark drinks as favorites and view them in a separate list.
  • Admins can create or delete custom drink bundles that appear alongside built-in recipes.
  • Users can rate and review each bundle.
  • Translate drink details into Spanish, French, or Italian.

Building

Open the project in Android Studio to build and run on an emulator or device.

Command Line Build

Run: ./gradlew assembleDebug

Troubleshooting

  • If you see an error about a missing wrapper jar, run gradle wrapper once to generate it.
  • If the Android Gradle plugin cannot be downloaded, ensure you have network access to Google and Maven repositories or install the dependencies manually.

Adding Videos

  1. Place tutorial videos in app/src/main/assets/videos/
  2. Update DrinkData.java with the file names

Bundles and Reviews

  1. Set Admin.IS_ADMIN to true in Admin.java to enable the Create Bundle button.
  2. Use the bundle editor to add name, ingredients, instructions, images, and videos for a new drink.
  3. Published bundles appear in the main list; select one to see details, delete it (admin only), and submit a rating or review.
Admin-only behavior

Admin actions (create/delete/publish) should not be accessible to non-admin users. Timeline Progress also includes admin-only events.

Timeline Progress: Use Case & Future Ideas

Overview

The Timeline Progress feature tracks a user’s journey through cocktail learning, bundle completion, and tasting milestones. It creates a chronological story of growth, highlights achievements, and helps users understand what they have learned, created, and reviewed over time.

This document covers the end-to-end use case, product expectations, UX flows, content strategy, data considerations, analytics, and future ideas to expand the feature.

Goals

  • Motivate learning by showing visible progress.
  • Surface context for favorites, bundles, and reviews chronologically.
  • Enable reflection with a historical view of practice and mastery.
  • Improve retention with reminders and achievable milestones.
  • Reinforce confidence by highlighting concrete wins and streaks.

Non-Goals

  • Replace core browsing or search flows.
  • Serve as a social feed (for now).
  • Provide real-time coaching (future enhancement).

Primary Use Case

User Story

As a bartender-in-training...

I want to see a timeline of my practice and learning so I can track which cocktails I’ve tried, which bundles I’ve completed, and how my skills evolve over time.

Key Outcomes

  • Users feel encouraged to continue learning and mixing.
  • Users can revisit past practice sessions or bundles for review.
  • Users can discover gaps (e.g., no progress for 2 weeks) and re-engage.
  • Users have a reliable, long-term record of activities and milestones.

Persona Examples

  • Beginner Enthusiast: Wants guidance and a sense of progression.
  • Home Bartender: Tracks favorites and tries new recipes every week.
  • Bundle Creator (Admin): Uses timeline to see how new bundles impact engagement.
  • Curious Explorer: Uses translations and notes to learn terminology.

User Journey Examples

Journey 1: First Week

  1. User favorites a Mojito.
  2. User views Mojito details and taps “Made It.”
  3. Timeline shows “First Cocktail Made” milestone.
  4. User completes a starter bundle.
  5. Timeline shows bundle completion and a 3-day streak.

Journey 2: Returning User

  1. User returns after 2 weeks of inactivity.
  2. Timeline highlights last completed drink and suggests a related recipe.
  3. User translates a recipe to Spanish and adds a tasting note.
  4. Timeline reflects translation usage and notes activity.

Journey 3: Admin Flow

  1. Admin publishes a themed bundle.
  2. Admin timeline includes “Bundle Published” event.
  3. Admin can see review submissions as follow-up events.

Timeline Events

Each entry represents a significant user action or milestone.

Core Event Types

Recipe Viewed Recipe Completed Bundle Started/Completed Favorite Added/Removed Review Submitted Language Translation Used Note Added

Admin-Only Events

Bundle Published Bundle Removed

Optional/Derived Events

Streak Achieved Milestone Reached Category Mastery

UX Flow

  1. Entry Point: Timeline is accessible from main navigation (e.g., “Progress”).
  2. Summary Header:
    • Total recipes tried.
    • Active streak length.
    • Last activity date.
    • Top category or most-made drink.
  3. Timeline Feed:
    • Scrollable list grouped by date.
    • Entries include icon, label, timestamp, and optional notes.
    • Color-coded event types (favorites, completions, reviews).
  4. Detail View:
    • Tap an event to revisit the drink or bundle detail screen.
    • Long-press to reveal actions (copy note, share, hide entry).
  5. Filters & Search:
    • Filter by event type (Completed, Reviewed, Favorited).
    • Search by drink name or bundle name.
  6. Empty State: Friendly CTA explaining how to start making cocktails.

Content & Copy Guidelines

  • Use celebratory but concise language (e.g., “You made your first Mojito!”).
  • Avoid shaming for inactivity; instead highlight the last positive action.
  • Keep entries skimmable: 1-line title + optional 1-line detail.
  • Respect localization; avoid slang that is hard to translate.

Data & Storage

Event Schema (Example)

{
  "event_id": "UUID",
  "user_id": "String",
  "event_type": "Enum",
  "timestamp": "ISO 8601",
  "entity_id": "Drink or bundle ID",
  "metadata": {
    "rating": "number (optional)",
    "translation_language": "string (optional)",
    "notes": "string (optional)",
    "streak_count": "number (optional)"
  },
  "source": "manual | auto | admin"
}

Event Generation Rules

  • Recipe Viewed: Only record if viewed > X seconds.
  • Recipe Completed: Requires explicit “Made It.”
  • Favorites: Emit on change to avoid duplicate events.
  • Reviews: Include rating score in metadata.
  • Translations: Include language code in metadata.

Storage Model

  • Local first (Room) with optional sync later.
  • Store minimal data in the event; read titles from source entities.
  • Cache lightweight event summaries for fast timeline rendering.

Aggregations

  • Weekly/Monthly Trends: Count of new completed recipes.
  • Streaks: Consecutive days with at least one completed recipe.
  • Top Categories: Most viewed spirit types or bundle themes.
  • Skill Velocity: Number of new recipes per week.

Progress States & Milestones

Progress is visualized using milestones:

  • First Cocktail Made
  • First Bundle Completed
  • 5/10/25/50 Cocktails
  • First Review Submitted
  • First Translation Used
  • First Tasting Note Added

Each milestone creates a timeline event and optionally unlocks a badge.

Edge Cases & Constraints

  • Offline Mode: Events are queued locally and synced later.
  • Duplicate Events: Prevent multiple “Made It” events within a short window.
  • Time Zone: Display events in user’s local time.
  • Admin vs User Events: Admin-only events appear only for admins.
  • Deleted Bundles: Timeline entries show “Archived Bundle” if source is deleted.
  • Device Migration: Show last sync time and explain missing history.

Metrics & Analytics

  • Timeline screen views per user.
  • Conversion from timeline entry → recipe detail view.
  • Event frequency by type (favorites vs completions).
  • Retention impact from timeline streaks.
  • Average time between events (cadence of engagement).
  • Completion rate from bundle start → bundle complete.

Accessibility Considerations

  • Icons paired with text labels.
  • High-contrast timeline markers.
  • Screen reader announces event type, date, and drink/bundle name.
  • Option to switch to “list view” for reduced motion.
  • Tap targets large enough for thumb navigation.

Privacy & Safety

  • Timeline is private by default; sharing is explicit.
  • Allow users to hide or delete entries.
  • Avoid recording sensitive data in metadata.
  • For admin events, avoid exposing private management data to non-admins.

Testing & QA Scenarios

  • Add timeline events for each core action (favorite, made it, review).
  • Confirm date grouping and time-zone conversion.
  • Validate filters and search across multiple event types.
  • Ensure offline actions sync correctly.
  • Verify empty state and first-use milestone.

Future Ideas

Short-Term Enhancements

  • Milestone Badges: Add a badge cabinet for achievements.
  • Sharing: Share milestones (“Made 10 cocktails!”) to social media.
  • Weekly Recap: Summary card for last 7 days.
  • Notes Field: Allow user to add tasting notes per recipe.
  • Contextual Tips: Show short tips related to recent events.

Mid-Term Enhancements

  • Personalized Suggestions: Use timeline data to recommend new cocktails.
  • Difficulty Levels: Track progress from beginner to advanced recipes.
  • Taste Profile Evolution: Show how preferences change over time (sweet → bitter).
  • Bundle Challenges: Timed challenges with a deadline.
  • Streak Rewards: Unlock recipes or visuals based on streaks.

Long-Term Vision

  • Collaborative Timelines: Friends can share progress or co-complete bundles.
  • Event Calendar Integration: Plan cocktail nights and sync reminders.
  • AI Practice Coach: Suggest practice routines based on gaps.
  • Vendor Partnerships: Offer ingredient kits tied to timeline milestones.
  • Community Benchmarks: Anonymous comparison for motivation.

Implementation Notes

  • Start with a local persistence layer (e.g., Room) and add optional sync later.
  • Reuse existing drink and bundle IDs for references.
  • Ensure timeline entries are created in all existing flows (favorites, reviews, bundle creation).
  • Keep the data model extensible for new event types.
  • Consider a background worker for aggregation and streak calculations.

Open Questions

  • Should users manually mark a recipe as “Made It” or infer from time-on-screen?
  • Do we allow users to delete or edit timeline entries?
  • How should timeline events appear if the user switches devices?
  • Should the timeline show failures (e.g., “attempted” recipes)?

Summary

The Timeline Progress feature turns the BartendingApp into a continuous learning journey, reinforcing engagement and surfacing meaningful milestones. It can start as a lightweight activity feed and evolve into a personalized coaching timeline with achievements, streaks, and community sharing.

Monetization (AdMob Banner)

To earn revenue by showing an ad at the bottom of the screen:

  1. Sign up for Google AdMob and create a banner ad unit.
  2. Add the Mobile Ads SDK to your module build.gradle:
implementation 'com.google.android.gms:play-services-ads:22.6.0'

AndroidManifest.xml

<uses-permission android:name="android.permission.INTERNET"/>
<uses-permission android:name="android.permission.ACCESS_NETWORK_STATE"/>

<meta-data
    android:name="com.google.android.gms.ads.APPLICATION_ID"
    android:value="ca-app-pub-2768983520806926~9920400584"/>

MainActivity Initialization

MobileAds.initialize(this);

activity_main.xml AdView

<com.google.android.gms.ads.AdView
    android:id="@+id/adView"
    android:layout_width="match_parent"
    android:layout_height="wrap_content"
    ads:adSize="BANNER"
    ads:adUnitId="ca-app-pub-2768983520806926/2820639760" />

Load an Ad

AdView adView = findViewById(R.id.adView);
adView.loadAd(new AdRequest.Builder().build());
